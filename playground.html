Featured Experiment

Experiment 01: Bash snapshot script

I'm learning Linux and Bash by building small utilities that give me immediate feedback. The goal is to make shell scripting feel practical, not academic.

Mini Project

system-snapshot.sh

bash linux automation logging

Instead of reading Bash tutorials forever, I'm building a small script that captures a health report for my machine and saves it as a timestamped log. It's practical, repeatable, and gives me a reason to combine real shell commands.

The script collects a few useful machine stats and writes them into a clean log file so I can inspect what's happening without retyping the same commands every time.

  • Creates a timestamp so each run is saved as its own report.
  • Captures uptime, disk usage, and the top memory-heavy processes.
  • Stores everything in a logs/ folder for later comparison.

Bash started feeling less abstract once I had a real script to shape. A few concepts clicked immediately when they were attached to something useful.

  • #!/usr/bin/env bash and chmod +x finally felt concrete.
  • $(...) command substitution made variables much more intuitive.
  • Pipes, redirects, and tools like ps and df make more sense together than in isolation.

Once the first version works, I want to treat it like a real mini tool instead of stopping at a one-time script.

  • Add flags like --quick and --full for different report sizes.
  • Archive older logs automatically so the folder stays clean.
  • Eventually schedule snapshots on a timer to compare system state over time.
#!/usr/bin/env bash
stamp="$(date +%Y-%m-%d_%H-%M-%S)"
report="logs/snapshot_$stamp.txt"

mkdir -p logs

{
  echo "host: $(hostname)"
  echo "uptime: $(uptime -p)"
  echo "disk:"
  df -h /
  echo "top memory processes:"
  ps -eo comm,%mem --sort=-%mem | head -n 6
} > "$report"

echo "saved report to $report"
